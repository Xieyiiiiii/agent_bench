# haystack_bm25 Analysis

## Why this passage was extracted

The CPU hot path of sparse retrieval is posting-list traversal with per-posting
score accumulation. The upstream projects include framework-level query
handling, tokenization, document objects, and optional sparse matrix layouts;
the benchmark extracts only the scoring and accumulation workload.

For CGRA benchmarking, this kernel deliberately stops after deterministic
posting traversal and fixed-point scoring. It does not build a corpus or compute
IDF at runtime; those preprocessing steps remain CPU/application concerns.

## C implementation target

Preserve:

- deterministic query term IDs;
- flat posting arrays and posting-list offsets;
- document-length normalized BM25Okapi score;
- deterministic synthetic `idf_q8[]` input;
- metadata filtering;
- active-document tracking before top-k.

Do not preserve:

- tokenizer and corpus preprocessing;
- `BM25Okapi._calc_idf()` and negative-IDF epsilon flooring;
- Python dictionaries or NumPy arrays;
- Haystack's full in-memory store implementation;
- bm25s compressed sparse row internals.

## CPU bottleneck modeled

This kernel stresses irregular memory access through posting lists, repeated
integer fixed-point arithmetic, filter branches, and final active-document
top-k selection. It is meant to model lexical retrieval scoring, not text
analysis.

The bottleneck is the accepted posting stream and score accumulation pattern,
not knowledge-base construction.

## Procedural implementation shape

The implementation should separate index setup, posting traversal, scoring, and
Top-K collection:

```text
main
  init_index
  init_query
  reset_state
  run_kernel
    score_posting_list_q8
      passes_filter
      bm25_term_score_q8
    collect_active_docs_into_topk
  checksum_result
  print_result
```

Use one `Bm25Index`, one `Bm25Query`, one `Bm25State`, and one `Bm25Counters`.
Avoid additional wrapper structures for postings, scores, or active flags.

## Behavior matching checks

- Empty posting lists must increment `EMPTY_TERMS`.
- `idf_q8[]` must be treated as deterministic input generated by `init_index()`,
  not recomputed by the scoring loop.
- Q8 scoring must use the fixed-point contract in `source_excerpt.md`, including
  int64 intermediates, truncation toward zero, and zero-denominator handling.
- Filtered postings must not affect scores or active flags.
- A document becomes active only after at least one accepted posting contributes
  to its score.
- Final Top-K must be collected from active documents, not from every document.
- Fixed-point variables that represent BM25 arithmetic must use `_q8` suffixes.
